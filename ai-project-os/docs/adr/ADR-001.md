# ADR-001: Use a repository-centered governance operating model

- Date: 2026-04-15
- Status: accepted
- Deciders: Director
- Technical area: governance

## Context

The project will be developed with multiple AI roles and frequent thread changes. Chat threads saturate quickly, and important information can be lost if decisions remain only in conversation history. The project also requires strong discipline because it is a scientific and engineering software effort where undocumented assumptions can damage both code quality and physical validity.

## Decision

The project will use a repository-centered governance model.

The repository, not chat history, is the authoritative project memory. Core rules, role boundaries, task packets, ADRs, handover packs, and status snapshots must live in repository documents. Work will be divided into explicit roles: Director, Product Physicist, Spec Lead, Developer, Auditor, and Documentation Steward. Protected-branch merges require review against a merge gate.

## Consequences

Benefits:
- Reduces loss of context during thread changes.
- Makes decisions reviewable and traceable.
- Keeps long-term project direction stable.
- Enables lower-cost models to work effectively through stronger process control.

Costs:
- Adds process overhead.
- Requires discipline from the human operator and all agents.
- Slows down ad hoc coding.

Trade-off:
The project accepts higher process overhead in exchange for lower long-term risk and better reproducibility.

## Alternatives considered

- Alternative 1: chat-centered workflow with one or two persistent threads.
- Alternative 2: single “super-agent” handling planning, coding, and review.
- Alternative 3: informal git workflow without explicit governance artifacts.

## Supersession

None.
